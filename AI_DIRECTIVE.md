## AI directive: using TapDB in `daylily-ursa`

This file is for future AIs/agents working in this repo.

### Non-negotiable identity rule

- Treat TapDB `euid` as the canonical identifier for new persisted Ursa entities.
- Do **not** mint UUIDs for persisted entities.
- Do **not** add new caller-generated `*_id` fields as canonical identity when TapDB can assign an `euid`.
- If you are adding new persistence for cluster creation, jobs, manifests, files, worksets, or similar records, use TapDB-generated `euid` values.

### What `uid` vs `euid` means here

- `uid`: TapDB internal/template-linked identifier.
- `euid`: external/business-facing canonical identifier generated by TapDB.
- In Ursa, new persisted objects should be referenced by `euid` unless an existing legacy business ID is already part of the product contract.

### Required create pattern

For new persisted records:

1. Open an explicit TapDB transaction with `session_scope(commit=True)`.
2. Ensure templates through the shared backend/manager.
3. Create the TapDB instance first.
4. Read back `row.euid`.
5. Use that `euid` as the canonical app-visible ID.
6. If the JSON payload needs the canonical ID embedded, update the stored payload after creation.

Do **not** do this:

- generate `uuid4()` for a new persisted record
- create a fake external ID first and treat TapDB as secondary storage
- bypass the shared TapDB abstractions with ad hoc persistence unless there is a proven gap

### Current repo examples to copy

- `daylib/workset_state_db.py:register_workset`
  - Creates a provisional TapDB workset instance.
  - Reads `ws.euid`.
  - Sets `workset_id = str(ws.euid)`.
  - Renames the instance and updates JSON payload with canonical `workset_id` and resolved prefix.

- `daylib/file_registry.py:register_file`
  - Creates the TapDB file row first.
  - If no file ID was provided, assigns `row.euid` to `registration.file_id` and `file_metadata.file_id`.
  - Writes the canonicalized payload back to TapDB.

- `daylib/file_registry.py:create_fileset`
  - Creates the fileset row first.
  - If no fileset ID was provided, assigns `row.euid` to `fileset.fileset_id`.
  - Writes the canonicalized payload back to TapDB.

### Abstractions to prefer

Use existing repo abstractions before adding new persistence code:

- `daylib.tapdb_graph.backend.TapDBBackend`
- `daylib.workset_state_db.WorksetStateDB`
- `daylib.file_registry.FileRegistry`
- `daylib.manifest_registry.ManifestRegistry`
- `daylib.workset_customer.CustomerManager`
- `daylib.biospecimen.BiospecimenRegistry`

If you need new persisted behavior for cluster-create flows, add it in the same style as the existing managers/registries instead of introducing raw one-off TapDB access patterns.

### Lookup/update rules

- Prefer lookups that accept either `euid` or legacy external IDs when the repo already supports both.
- Do not assume older `*_id` fields are canonical just because they exist in payloads.
- When extending existing entities, preserve current compatibility behavior, but do not introduce new UUID-era identity schemes.

### Transaction and audit rules

- Use explicit transaction ownership via `session_scope(commit=True)` for writes.
- Preserve TapDB audit attribution behavior; do not work around the connection/session conventions.
- Use the shared backend so session username and environment wiring stay correct.

Reference: `docs/ursa_tapdb_refactor_design.md`, especially “TapDB conventions to adopt”.

### Soft delete and status rules

- Follow TapDB soft-delete semantics already used in the repo.
- Prefer marking records deleted/inactive through existing model conventions rather than hard-deleting rows.
- Keep `bstatus` aligned with lifecycle state when extending persisted objects.

### Namespace/environment rules

- Ursa runs TapDB in strict namespace mode.
- Use configured environment resolution; do not hardcode DB connection details.
- `TapDBBackend` already enforces the expected namespace/environment behavior and should remain the default entrypoint.

### Legacy-ID caveat

Some legacy business IDs still exist, especially in customer-facing payloads such as `customer_id`.

That does **not** justify introducing new UUID-based IDs for new persisted object types.

Rule of thumb:

- existing legacy business ID required by current product contract: preserve it
- new persisted object identity: use TapDB `euid`

### Specific instruction for upcoming cluster-create work

- If cluster creation needs persisted job/request/state records, create them in TapDB first and use returned `euid` values as canonical IDs.
- Do not reintroduce UUID-based job IDs, cluster-request IDs, or workflow IDs.
- Mirror the workset/file/fileset pattern already established in this repo.
